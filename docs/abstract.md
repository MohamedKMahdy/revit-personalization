# Abstract

## Agent-Augmented BIM Log Mining for Personalized Action Generation

---

Building Information Modelling (BIM) authoring tools such as Autodesk Revit are central to contemporary design and construction workflows. Yet despite decades of development, these platforms remain fundamentally interaction-intensive: designers must repeatedly execute long chains of low-level micro-operations to realise even simple design intents. Chief among these repetitive sequences is what this thesis terms the *Custom Element Instantiation loop* — the act of placing a typed element family (a door, a structural column, a lighting fixture), subsequently setting a constellation of its parameters (fire rating, mark number, classification code, dimension), and finally applying an annotation tag to contextualise it within the drawing. This sequence is enacted dozens of times per project session, yet it remains entirely manual, click-heavy, and immune to the scripted automation that governs formal workflows. It is not that designers are unaware of the repetition; it is that no existing tool learns the pattern from observation and converts it into reusable automation without requiring the designer to specify it explicitly.

Recent advances in generative artificial intelligence have introduced new categories of assistance into BIM authoring environments. Autodesk Revit 2027 ships with an integrated Autodesk Assistant — an AI-powered chat panel connected to the active model via the Model Context Protocol (MCP) — and exposes a public MCP server that allows external agents to query model state through natural language. Community bridges have demonstrated that large language models can execute discrete commands against a live Revit model, enabling what practitioners describe as one-click modelling tasks. These developments confirm the technical feasibility of AI-driven automation in BIM. However, they share a common limitation: they act when prompted by the user and operate on tasks that are fully specified at the time of invocation. They do not observe, learn, or adapt. The personalised, implicit, subconscious micro-routines that define each designer's individual working style — and that constitute the dominant source of productivity loss at the micro-interaction level — remain entirely unaddressed.

This thesis identifies four specific research gaps that motivate the proposed system. First, existing BIM log mining and command recommendation systems focus on predicting the next single command from a history of previous commands, treating the authoring session as a flat sequence of discrete events. They do not synthesise multi-step, executable automation from the detected patterns; the bridge between sequence modelling and automatic generation of reusable scripts has not been built. Second, commercial AI assistants such as the Autodesk Assistant provide personalisation in terms of context from the current model and explicit user prompts, but they do not model implicit behavioural habits inferred from interaction telemetry. The Custom Element Instantiation loop — a pattern that manifests at the sub-command level through coordinated placements, parameter assignments, and annotations — is invisible to systems that operate at the level of natural language requests. Third, existing command recommendation models are trained centrally on large aggregated datasets and are not adapted per-user from small local samples. Applying few-shot meta-learning principles to individual BIM authoring logs remains unexplored, particularly under the privacy and intellectual property constraints that prevent firms from sharing detailed project telemetry externally. Fourth, all current MCP-based agents in the BIM domain are reactive: they respond to explicit prompts. No existing work demonstrates a system that continuously monitors a user's interaction stream, detects the emergence of a repeated routine in real time, and proactively proposes to automate it — without the user having to recognise or name the pattern themselves.

The thesis proposes a system that addresses all four gaps through a three-component, locally-executing architecture. The first component is a native C# Revit add-in, RevitLogger, which runs continuously inside the Revit 2027 process and observes every authoring transaction at the API event level. The second component is a Python-based MCP server that reads the add-in's output, detects candidate routines through a structural episode-grouping algorithm, and exposes both the raw evidence and the detected patterns as MCP resources and tools. The third component is a multi-agent orchestrator built on the Anthropic Claude API, comprising a Pattern Agent that extracts generalisable workflow motifs from a small number of recorded examples and a Macro Agent that translates those motifs into executable shortcut configurations.

The RevitLogger add-in subscribes to Revit's DocumentChanged event — which fires precisely once per committed transaction, the atomic unit of user intent — and intercepts every element placement, parameter modification, and annotation tag operation that falls within the set of component-level authoring categories defined by the Jang et al. (2023) lexicon. The logging schema implemented by the add-in directly follows the enhanced BIM logging framework proposed by Jang and Lee (2023), which identified that raw Revit journal files lack the semantic context necessary for reproducible workflow analysis. Accordingly, every captured record carries a transaction identifier that groups all element modifications arising from a single user intent, the undo-stack label that provides semantic context without requiring natural language processing, and before-and-after parameter value pairs that enable the detection of what changed rather than merely which element was touched. The schema is further enriched with spatial and temporal context fields — level name, phase name, view name, and view type — that Jang and Lee (2023) demonstrated to be essential for characterising the preconditions under which a routine is executable. The add-in filters out all auto-computed parameters (area, volume, phase-created, workset, and room-derived fields) to ensure that only parameters reflecting genuine user intent are presented to the downstream agents. Log files are written in JSONL format — one JSON object per line, UTF-8 without BOM — to a local directory, using a thread-safe asynchronous queue that ensures the Revit UI thread is never blocked by disk I/O.

The Python MCP server implements an episode-grouping detection algorithm that treats the element identifier as the primary key linking all records belonging to the same Custom Element Instantiation episode. For each element observed in the log files, the algorithm reconstructs the complete episode — Place record, followed by any number of SetParam records, followed optionally by a Tag record — and computes a structural signature encoding the element category, the family name, and the ordered sequence of action types and parameter names. Episodes with identical signatures constitute instances of the same routine. Signatures appearing in two or more episodes become CandidateRoutine objects, ranked by repetition count. The detection algorithm operates across all session files simultaneously, enabling cross-session pattern recognition. The server exposes the detected routines as MCP resources accessible via the URI scheme logs://candidate_routines and logs://routine/{id}/examples. It also exposes five tools: analyze_pattern for validating and preparing sequences for agent consumption; generate_command for persisting a learned shortcut configuration to disk; execute_revit_command for triggering shortcut execution inside the live Revit model; query_model for issuing read-only queries to the Autodesk Public MCP Server; and list_shortcuts for enumerating saved automation. The server runs on port 3100 using Server-Sent Events transport and is designed to be registered as an additional endpoint within the Autodesk Assistant, allowing users to interact with the personalisation system through natural language inside Revit's native interface.

The multi-agent orchestrator separates the cognitive demands of the pipeline into two specialised roles. The Pattern Agent uses the claude-opus-4-7 model with extended thinking enabled at an eight-thousand-token reasoning budget. It receives k recorded examples of a candidate routine and is tasked with extracting a generalised motif: the invariant action sequence, the classification of each parameter value as constant (identical across all examples) or variable (differing across examples and requiring user input at execution time), and the preconditions necessary for the routine to be applicable. The use of extended thinking is deliberate: the classification of a parameter as constant when it is actually variable would produce a shortcut that silently overwrites user-intended variation, a failure mode that is both subtle and consequential. Extended reasoning provides the depth required to distinguish genuine invariance from coincidental consistency across a small sample. The Macro Agent uses the claude-sonnet-4-6 model to translate the extracted motif into an ordered sequence of tool calls — place_element, set_parameter, create_annotation_tag — with constant parameter values resolved to their literal values and variable parameters replaced by runtime placeholders that are resolved from user input when the shortcut executes.

A key architectural finding of this thesis concerns the capability boundaries of the Autodesk Public MCP Server, which was released as a Tech Preview alongside Revit 2027. Contrary to the system's original design assumption, the public server is confirmed read-only in its current release: it supports model queries across six tool groups (model queries, sheet management, room management, schedules, exports, and element operations) but cannot create, modify, or delete elements. This finding necessitated a redesign of the execution layer. Rather than routing shortcut execution through the public server, the system uses a file-based inter-process communication protocol: the Python layer writes a pending_execution.json request file to a shared local directory; the C# add-in's ShortcutRunner component, implemented using a FileSystemWatcher, detects the request, reads the corresponding ShortcutConfig specification, opens a Revit API transaction, executes each placement, parameter assignment, and tagging step in sequence, and writes an execution_result file that the Python layer reads to confirm completion. This design is both more reliable and more deterministic than execution through an external server: it requires no network dependency, produces identical results on every invocation for identical inputs, and maintains the project's strict local-processing constraint.

The evaluation methodology follows the two-metric framework specified in the thesis proposal. The sample efficiency experiment measures Pattern Agent accuracy as a function of k, the number of example episodes provided, by scoring the extracted motif against the ground-truth episode structure on two dimensions: step-match accuracy, the fraction of ground-truth action steps correctly predicted, and parameter coverage, the fraction of ground-truth SetParam steps whose parameter names appear in the extracted motif. The process acceleration experiment measures task completion time and action count for selected Custom Element Instantiation workflows under manual versus assisted conditions. The evaluation harness, implemented in eval/run_experiment.py, automates the sample efficiency experiment across configurable k values and repetitions, recording token usage and latency alongside accuracy scores, and writing results to a structured CSV file suitable for direct inclusion in the thesis evaluation tables.

The system represents several concrete contributions to the research literature. It is the first demonstrated implementation of few-shot, per-user BIM workflow personalisation from local interaction logs, operating entirely without cloud data upload and without requiring the user to label, name, or otherwise specify the routines to be learned. It provides empirical confirmation of the read-only limitation of the Autodesk Public MCP Server, a factual finding about the current state of the ecosystem that informs how practitioners and researchers should position MCP-based BIM automation. It demonstrates the registration of a custom personalisation MCP server as an additional endpoint within the Autodesk Assistant, creating a natural language interface to a locally-executing learning system that is embedded within Autodesk's own AI infrastructure rather than competing with it. And it implements the enhanced BIM logging schema of Jang and Lee (2023) in a production-quality Revit 2027 add-in, validated against the API changes introduced in the 2027 release cycle, providing a reusable logging foundation for future BIM interaction research.

The hypothesis advanced by this thesis — that few-shot meta-learning principles applied to local user telemetry via an MCP-augmented multi-agent system can accurately discover personal Custom Element Instantiation motifs and autonomously generate executable Revit shortcuts — is operationalised in a working end-to-end system. From the moment a designer opens a project in Revit, their authoring actions are captured and structured. Once a pattern has repeated, the orchestrator extracts its invariant logic through agent-based reasoning, generates a named shortcut, and makes that shortcut available for one-click execution — all on the local machine, with no project data leaving the network, and with the full expressiveness of a foundation model's reasoning applied to the problem of understanding what the designer is trying to do.
